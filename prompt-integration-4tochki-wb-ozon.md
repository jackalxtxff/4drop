# Промпт: система интеграции шинного поставщика 4tochki с Wildberries и Ozon

> Вставь этот промпт целиком в начало новой сессии. Он задаёт контекст, требования
> и границы задачи. Прежде чем писать код — пройди раздел «Что уточнить до старта».

---

## Роль и цель

Ты — инженер, проектирующий и собирающий SaaS-подобную систему, которая связывает
B2B-поставщика шин и дисков **4tochki (Форточки, `b2b.4tochki.ru`)** с кабинетами продавца
на **Wildberries** и **Ozon**. Модель бизнеса — по сути дропшиппинг: товар физически лежит
на складах 4tochki, продажа идёт через маркетплейсы, заказ передаётся поставщику.

Система должна закрывать три сценария:

1. **Создание карточек товаров** в кабинетах на обеих площадках на основе данных 4tochki.
2. **Синхронизацию остатков и цен** — подтягивать из 4tochki, применять наценку, пушить в WB и Ozon.
3. **Передачу заказов** — по мере поступления заказов на маркетплейсах автоматически
   создавать заказ (бронь) в 4tochki.

К этому добавляется **веб-интерфейс** для управления (описан ниже) — это первоочередная часть.

---

## Внешние API (три стороны)

### 4tochki (сторона поставщика) — ВХОД ПО ЛОГИНУ/ПАРОЛЮ
- Каталог с описаниями, брендами, моделями, картинками (выгрузка + веб-сервис).
- Остатки в реальном времени по нескольким складам (склады выбираются).
- Проценка: цена + наличие по коду производителя **CAE** (это ключ товара).
- Создание заказа поставщику, бронирование, статусы заказа.
- Подключение для B2B-клиентов бесплатное.
- **НЕИЗВЕСТНО и требует подтверждения:** протокол (SOAP/WSDL или REST/JSON), точные
  имена методов, поля запроса/ответа, лимиты. Не выдумывай сигнатуры — запроси у
  пользователя текст справки `b2b.4tochki.ru/Help` или WSDL и построй клиент по ним.
  Авторизация в API — по логину/паролю B2B-кабинета.

### Wildberries (сторона продавца) — ВХОД ПО API-КЛЮЧУ (токен/OAuth 2.0)
- **Content API** — создание/редактирование карточек.
- **Prices & Discounts API** — цены и скидки.
- **Marketplace API** — остатки FBS + сборочные задания (= заказы).
- Схемы: FBS / DBS / C&C / FBW. Для дропшипа целимся в FBS или realFBS-аналог.
- **Важно (2026):** остатки передаются по `chrtId`, а не по `sku`. Учитывай в маппинге.

### Ozon (Seller API) — ВХОД ПО `Client-Id` + `Api-Key`
- **Product API** — создание/обновление товаров (проходит модерацию, задержка до нескольких дней).
- **Prices & Stocks** — `v4/product/info/stocks` и методы цен.
- **Posting API** — заказы FBS/realFBS (`v3/posting/fbs/unfulfilled/list`, `v3/posting/fbs/list`).
- Push-уведомления о новых заказах (использовать для near-real-time).

## Веб-интерфейс (первоочередная часть)

### Аутентификация приложения
- Вход в саму систему по **логину и паролю** (собственные учётки пользователей приложения).
- Хранение паролей — хеш (argon2/bcrypt). Сессии/JWT. Роли можно заложить на будущее
  (admin/operator), но на старте достаточно одной роли.

### Раздел «Поставщики» (Suppliers)
- **Поставщик — центральная сущность системы.** Пользователь создаёт поставщика
  (название, комментарий), и уже к нему привязываются доступы к API всех трёх площадок
  (4tochki + Wildberries + Ozon). Все товары, маппинги, заказы и синхронизации
  относятся к конкретному поставщику.
- CRUD поставщиков: создать, переименовать, отключить/удалить.
- Возможность завести **несколько поставщиков**, у каждого — свой набор доступов и свой каталог.
- В интерфейсе — переключатель активного поставщика (весь остальной UI показывает данные
  выбранного поставщика). Раздел «Подключения» и вкладка «Товары» работают в контексте
  выбранного поставщика.

### Раздел «Подключения» (Connections / Credentials) — в контексте выбранного поставщика
Три блока для ввода и хранения доступов, **привязанных к текущему поставщику:**
1. **4tochki:** поля `login`, `password`, выбор активных складов (список складов
   подтягивается из API после успешного подключения). Кнопка «Проверить подключение».
2. **Wildberries:** поле для API-ключа (или набора ключей по категориям: content, prices,
   marketplace) либо OAuth-подключение. Кнопка «Проверить».
3. **Ozon:** поля `Client-Id` и `Api-Key`. Кнопка «Проверить».

Требования к блоку:
- Секреты хранить **зашифрованными** (не в открытом виде), в БД — только шифртекст;
  ключ шифрования — из окружения/секрет-хранилища, не в репозитории.
- В интерфейсе показывать статус подключения (ок / ошибка / не настроено) и маскировать
  секреты (последние 4 символа).
- Поддержать несколько наборов доступов на будущее (мультиаккаунт), но на MVP — один набор на площадку.

### Вкладка «Товары» (главный экран)
- **Таблица товаров, выгруженных из 4tochki.** Колонки минимум: CAE, бренд, модель,
  наименование, размерность/характеристики, склад(ы), остаток, закупочная цена, картинка (превью),
  статус интеграции (не интегрирован / на WB / на Ozon / на обоих).
- **Поиск и фильтры:** по бренду, сезону, размерности, наличию, цене, статусу интеграции.
- **Множественный выбор** строк (чекбоксы + «выбрать всё по фильтру»).
- **Действие «Интегрировать»:** для выбранных товаров выбрать целевые площадки (WB / Ozon / обе)
  и запустить создание карточек. Показать прогресс и результат (успех / на модерации / ошибка с причиной).
- Кнопка «Обновить каталог из 4tochki» (ручной запуск выгрузки) + отметка времени последней синхронизации.
- Пагинация/виртуализация — каталог крупный (десятки тысяч позиций).

### (Желательно) Вкладки «Заказы» и «Синхронизация»
- **Заказы:** лента заказов с маркетплейсов, их статус, связанный заказ/бронь в 4tochki, ошибки.
- **Синхронизация:** расписания и логи задач обновления остатков/цен, лог ошибок API.

## Модель данных (ориентир)
- `users` — учётки веб-приложения.
- `suppliers` — **поставщик, центральная сущность:** название, статус, владелец.
  К нему привязаны доступы, каталог, заказы и синхронизации. Большинство таблиц ниже
  имеют внешний ключ `supplier_id`.
- `credentials` — доступы к 4tochki/WB/Ozon (зашифровано), статус, выбранные склады;
  **привязаны к `supplier_id`** (один набор на площадку в рамках поставщика).
- `products` — товары из 4tochki (ключ CAE) в рамках `supplier_id`, кэш каталога:
  атрибуты, картинки, склад, остаток, цена.
- `product_links` — маппинг CAE ↔ WB (`nmID`/`chrtId`/`barcode`) ↔ Ozon (`offer_id`/`product_id`),
  плюс статус интеграции и целевые площадки (в рамках `supplier_id`).
- `pricing_rules` — правила наценки (процент/фикс/округление), при необходимости по категориям и поставщику.
- `orders` — заказы с MP, связь с заказом/бронью 4tochki, статусы (в рамках `supplier_id`).
- `sync_jobs` / `logs` — задания синхронизации и журнал ошибок (в рамках `supplier_id`).

## Фоновые процессы (после MVP интерфейса)
- Периодическая выгрузка каталога 4tochki → обновление `products`.
- Синхронизация остатков и цен: 4tochki → наценка → WB (prices + stocks по `chrtId`) и Ozon (prices + stocks).
- Приём заказов: polling сборочных заданий WB + постингов Ozon (лучше webhook/push) →
  маппинг SKU→CAE → бронь/заказ в 4tochki → обновление статуса.
- Обработка ошибок, ретраи, соблюдение rate limits каждой площадки.

## Критичные ограничения (заложить в дизайн, не игнорировать)
- **Маркировка «Честный знак».** Шины — товар обязательной маркировки. Выяснить у 4tochki
  из договора, кто и когда передаёт коды КИЗ и как это встраивается в FBS/realFBS. Это способно
  изменить всю схему — считать блокирующим вопросом на этапе проектирования.
- **Оверселл.** Остаток 4tochki общий и живой, его разбирают все их клиенты. Нужен буфер
  остатков и мгновенная бронь при заказе, иначе штрафы за неотгрузку.
- **Маппинг SKU** (CAE ↔ WB ↔ Ozon) — стержень задач 2 и 3, ошибки = продажа не того товара.
- **Модерация карточек** на WB и Ozon: задержки, отклонения — интерфейс должен показывать статус.
- **Rate limits** и мелкие изменения форматов API (как переход WB на `chrtId`) — мониторинг и логи.
- **Модель фулфилмента** (везти к себе и FBS против realFBS/DBS напрямую от 4tochki) — определяет SLA.
- **Возвраты и гарантия** — отдельный процесс, продумать юридически.

## Технологический стек (обязательные требования)
- **Всё работает из Docker.** Каждый сервис — в контейнере, запуск одной командой
  `docker compose up`. Отдельные сервисы минимум: бэкенд (API), фронтенд, PostgreSQL,
  воркер фоновых задач и брокер очереди (Redis/RabbitMQ). Конфигурация — только через
  переменные окружения (`.env`), секреты не хардкодить и не коммитить. Приложить
  `Dockerfile` для каждого сервиса и `docker-compose.yml`; для фронтенда — сборка и раздача
  статики (или dev-режим для разработки). Учесть тома для данных БД и удобный локальный запуск.
- Бэкенд: Python (FastAPI) или Node (NestJS). Очереди/планировщик для фоновых задач.
- БД: PostgreSQL. Шифрование секретов на уровне поля.
- **Фронтенд: React + Tailwind CSS.** Вёрстка на Tailwind (utility-классы), без сторонних
  тяжёлых UI-китов без необходимости. Таблица товаров — с виртуализацией (для больших
  каталогов) и серверной фильтрацией/пагинацией.

## Порядок работы
1. Собери и подтверди ответы на вопросы выше.
2. Сначала — веб-интерфейс: auth, раздел «Подключения», вкладка «Товары» с таблицей и выбором.
